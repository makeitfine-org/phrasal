What happens when a VM goes down
---

  ┌──────────────────────────┬───────────────────────────────────────────────────────────────┬───────────────────────────┐
  │         Scenario         │                         What happens                          │      Auto-recovery?       │
  ├──────────────────────────┼───────────────────────────────────────────────────────────────┼───────────────────────────┤
  │ VM2 (worker) dies        │ Pods on VM2 get rescheduled to VM1                            │ Yes, automatic (~1-5 min) │
  ├──────────────────────────┼───────────────────────────────────────────────────────────────┼───────────────────────────┤
  │ VM1 (control plane) dies │ Entire cluster is dead — no rescheduling, no kubectl, nothing │ No                        │
  ├──────────────────────────┼───────────────────────────────────────────────────────────────┼───────────────────────────┤
  │ Both die                 │ Everything down                                               │ No                        │
  └──────────────────────────┴───────────────────────────────────────────────────────────────┴───────────────────────────┘

  VM2 dies — K8s handles it

  1. Control plane on VM1 detects VM2 is unreachable (~40 seconds by default)
  2. Marks VM2 as NotReady
  3. After 5 minutes (default pod-eviction-timeout), reschedules pods to VM1
  4. Your app comes back on VM1 automatically

  But — VM1 needs enough resources to run everything (control plane + all your pods). With 2 CPU / 12GB this is tight but
  doable.

  VM1 dies — you're stuck

  Single control plane = single point of failure. K8s cannot fix this because the brain is dead. You'd need to:

  1. SSH into VM1 (once OCI restores it) or rebuild it
  2. Restart kubelet / control plane components
  3. Pods come back after control plane recovers

  To fix this properly you'd need 3 control planes (etcd quorum requires odd numbers) — not realistic on 2 VMs.

  What you should activate

  K8s self-healing is on by default, but configure these:

  # Pod disruption budget — keeps at least 1 backend running
  apiVersion: policy/v1
  kind: PodDisruptionBudget
  metadata:
    name: backend-pdb
  spec:
    minAvailable: 1
    selector:
      matchLabels:
        app: backend

  # Liveness + readiness probes — K8s restarts unhealthy pods
  livenessProbe:
    httpGet:
      path: /actuator/health
      port: 8080
    initialDelaySeconds: 30
    periodSeconds: 10
  readinessProbe:
    httpGet:
      path: /actuator/health/readiness
      port: 8080
    initialDelaySeconds: 10
    periodSeconds: 5

  # Run 2 backend replicas across both nodes
  replicas: 2
  affinity:
    podAntiAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 100
          podAffinityTerm:
            labelSelector:
              matchLabels:
                app: backend
            topologyKey: kubernetes.io/hostname

  Reasons VMs go down

  ┌────────────────────────────┬──────────────────────────────────────────────────────────┐
  │           Reason           │                    Likelihood on OCI                     │
  ├────────────────────────────┼──────────────────────────────────────────────────────────┤
  │ OCI maintenance/reboot     │ Rare, usually scheduled with notice                      │
  ├────────────────────────────┼──────────────────────────────────────────────────────────┤
  │ OS crash / kernel panic    │ Very rare                                                │
  ├────────────────────────────┼──────────────────────────────────────────────────────────┤
  │ Disk full                  │ Common if you don't monitor — Docker images fill up fast │
  ├────────────────────────────┼──────────────────────────────────────────────────────────┤
  │ OOM (out of memory)        │ Possible if no resource limits set                       │
  ├────────────────────────────┼──────────────────────────────────────────────────────────┤
  │ You accidentally delete it │ It happens                                               │
  └────────────────────────────┴──────────────────────────────────────────────────────────┘

  Bottom line

  K8s auto-heals worker failures. It cannot auto-heal control plane failure with only 1 control plane node. For 100 users on
  a learning app, this is perfectly acceptable — just set up the health probes and you're good.

  What does it mean reschedule

● Reschedule means K8s automatically moves your app from the dead VM to a healthy one.

  Example: VM3 dies

  BEFORE                          AFTER (automatic, ~1-5 min)

  VM1 (control plane)             VM1 (control plane)
    - watches everything            - detects VM3 is gone
                                    - creates PostgreSQL pod on VM2

  VM2 (worker)                    VM2 (worker)
    - backend                       - backend
    - frontend                      - frontend
                                    - PostgreSQL ← moved here

  VM3 (worker)                    VM3 💀 dead
    - PostgreSQL

---
  K8s doesn't literally "move" the container — it starts a brand new pod on the surviving VM from the same image. The old
  pod on the dead VM is gone.
---

  Important for PostgreSQL

  This means the pod is new — if your database data was stored on VM3's local disk, it's lost. You need one of:

  ┌─────────────────────────────────┬───────────────────────────────────────────────┐
  │        Storage approach         │           Data survives reschedule?           │
  ├─────────────────────────────────┼───────────────────────────────────────────────┤
  │ Local disk (default)            │ No — data dies with VM                        │
  ├─────────────────────────────────┼───────────────────────────────────────────────┤
  │ OCI Block Volume (network disk) │ Yes — K8s reattaches it to the new pod on VM2 │
  ├─────────────────────────────────┼───────────────────────────────────────────────┤
  │ OCI Block Volume replicated     │ Yes + survives disk failure                   │
  └─────────────────────────────────┴───────────────────────────────────────────────┘

  So for your PostgreSQL, you must use a PersistentVolume backed by OCI Block Volume. Otherwise rescheduling "saves" the app
  but loses all your data.

  apiVersion: v1
  kind: PersistentVolumeClaim
  metadata:
    name: postgres-data
  spec:
    accessModes:
      - ReadWriteOnce
    resources:
      requests:
        storage: 50Gi
    storageClassName: oci-bv    # OCI Block Volume

  This way when PostgreSQL reschedules to VM2, the same disk gets reattached and all data is intact.

● OCI Block Volume — Free Tier

  Yes, it's free. OCI gives you 200GB total block storage in the Always Free tier. This is shared across all your VMs (boot
  volumes + extra volumes).

  Typical usage:

  ┌────────────────────────┬────────────────┬─────────────────────────┐
  │         Volume         │      Size      │         Purpose         │
  ├────────────────────────┼────────────────┼─────────────────────────┤
  │ VM1 boot volume        │ 47GB (minimum) │ OS + control plane      │
  ├────────────────────────┼────────────────┼─────────────────────────┤
  │ VM2 boot volume        │ 47GB           │ OS + containers         │
  ├────────────────────────┼────────────────┼─────────────────────────┤
  │ VM3 boot volume        │ 47GB           │ OS + containers         │
  ├────────────────────────┼────────────────┼─────────────────────────┤
  │ PostgreSQL data volume │ 50GB           │ Database data           │
  ├────────────────────────┼────────────────┼─────────────────────────┤
  │ Total                  │ ~191GB         │ Within 200GB free limit │
  └────────────────────────┴────────────────┴─────────────────────────┘

  When to create it

  Step 1 — Create VMs (OCI Console or CLI)
  Each VM automatically gets a boot volume. Nothing extra to do here.

  Step 2 — Create the Block Volume separately

  OCI Console → Storage → Block Volumes → Create Block Volume
    - Name: phrasal-postgres-data
    - Size: 50GB
    - Availability Domain: same as VM3 (important!)

  Step 3 — Don't attach it to any VM manually. Let K8s handle it via the OCI CSI driver. You just create the
  PersistentVolumeClaim in K8s and it attaches/detaches automatically — including during rescheduling.

  How it works during reschedule

  VM3 dies
    ↓
  K8s detaches block volume from VM3
    ↓
  K8s attaches block volume to VM2
    ↓
  K8s starts PostgreSQL pod on VM2 with same volume
    ↓
  PostgreSQL starts, reads existing data — nothing lost

  One caveat

  Block volumes are availability-domain specific on OCI. If your VMs are in different availability domains, the volume can't
  reattach. So when creating VMs:

  Put all 3 VMs in the same Availability Domain.

  This is the most common mistake — if you skip this, rescheduling works but the disk can't follow.
